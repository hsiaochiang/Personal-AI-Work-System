# Decision Log

## 2026-03-12

### 決策：本專案的核心目標定版

- 決策內容：以建立個人 AI 工作系統為主軸，重點包含核心知識庫、專案記憶、工作流、SKILL 演化、降低啟動摩擦
- 決策理由：這些目標最貼近使用者真正要解決的問題，不再把重心放在提示詞管理本身
- 影響範圍：整體專案方向

### 決策：規劃順序採用「目標 -> 原則 -> 策略 -> 運作方式 -> MVP -> 回顧迭代」

- 決策內容：先完成規劃框架，再進入具體做法
- 決策理由：降低做到一半發現方向不合適的風險
- 影響範圍：規劃流程與後續討論順序

### 決策：第一階段採跨工具思維設計，但先從 VS Code / Copilot Chat 做 MVP

- 決策內容：架構上以跨工具共用知識為前提，實作上先處理最容易取得的對話紀錄來源
- 決策理由：兼顧長期方向與短期可驗證性
- 影響範圍：第一階段資料來源與 MVP 範圍

### 決策：第一階段先做知識沉澱，不先做執行助手

- 決策內容：優先建立對話到結構化知識的沉澱流程
- 決策理由：先建立穩定記憶層，後續 agent 或 workflow 才有可靠基礎
- 影響範圍：第一階段系統設計重心

### 決策：第一階段以專案為核心，採半自動模式

- 決策內容：先累積專案記憶，由系統提議沉澱內容，再由人工確認寫入
- 決策理由：最符合目前痛點，也較容易兼顧品質與使用成本
- 影響範圍：文件結構、流程設計與沉澱方式

### 決策：專案文件結構採「每個專案一個資料夾，按主要類型拆檔」

- 決策內容：每個專案保留一份入口檔，並以主要記憶類型分檔管理
- 決策理由：比單一大檔更容易長期維護，也比過度細分更容易開始使用
- 影響範圍：專案檔案組織方式

### 決策：保留規劃串與實作串分離，但不再把 git 分支與對話長度直接綁定

- 決策內容：對話切分依任務目標與上下文邊界決定；git 分支則依程式變更邊界決定
- 決策理由：降低長對話帶來的漂移，同時避免為了上下文過長而開出過多分支
- 影響範圍：後續對話策略、實作節奏與分支管理方式

### 決策：本專案的長期成果應該包含 UI，但第一階段先不做完整介面

- 決策內容：本專案長期應收斂成一個個人 AI 工作台；第一階段先以 markdown、流程與模板驗證核心工作流
- 決策理由：先把高頻工作流跑通，再設計真正有價值的 UI，能避免過早做出華而不實的介面
- 影響範圍：產品方向、階段規劃與後續實作順序

### 決策：未來 UI 以 Project Hub、Workspace、Handoffs、Memory Review 為第一版核心畫面

- 決策內容：未來若開始做 UI，第一版最小可用畫面集合為 Projects、Workspace、Handoffs、Memory Review
- 決策理由：這四個畫面最直接對應目前已定義的核心工作流
- 影響範圍：UI 規劃與未來 MVP 範圍

## 2026-03-14

### 決策：V1 採 4 週穩健版路線，先完成可用的工作流，再追求更多自動化

- 決策內容：V1 先以 4 週穩健版 roadmap 推進，優先完成手動流程驗證、半自動提取 MVP、真實專案驗證與 V1 收斂
- 決策理由：本專案的價值核心在於工作流可持續使用與沉澱品質，過早追求速度容易做出能跑但不好用的系統
- 影響範圍：V1 時程規劃、實作順序與功能邊界

### 決策：專案文件依功能重組為 planning、memory、workflows、templates、product、roadmap、references

- 決策內容：根目錄保留 `README.md` 作為入口，其餘文件依用途分類到對應資料夾
- 決策理由：在進入 Phase 1 前先把資訊結構整理好，方便後續在 VS Code 另開 session 時快速找到正確文件
- 影響範圍：專案檔案結構、文件導覽方式與後續維護

### 決策：除 V1 外，另建立全專案 roadmap 作為長期演進視圖

- 決策內容：以 `roadmap/project-roadmap.md` 追蹤 Phase 0 到 V3 的整體演進，而 `roadmap/v1-roadmap.md` 專注第一版
- 決策理由：V1 已不足以承接整個專案願景，需要有更長期的版本視角來對齊後續投資方向
- 影響範圍：專案規劃層級、版本管理與後續決策討論
