# Preference Rules

## 使用原則

- 規則應使用白話表達，避免抽象術語造成歧義
- 規則只記錄穩定、可重用的偏好，不收錄一次性要求
- 不確定是否穩定的偏好，先列為候選

## 已確認規則

### 規則 1

- 規則：在目標與做法之間，應明確規劃原則、策略與運作方式
- 說明：使用者希望在規劃階段投入較高比重，避免做到一半才發現不合適
- 來源依據：本專案初期討論
- 狀態：已確認

### 規則 2

- 規則：命名應白話、直覺，避免造成誤解
- 說明：例如「適配層」與「操作模型」都因為不夠直覺而被要求改名
- 來源依據：本專案初期討論
- 狀態：已確認

### 規則 3

- 規則：系統設計應優先低摩擦與可持續使用
- 說明：若流程太重，即使理論完整，也不適合作為核心工作方式
- 來源依據：設計原則定版
- 狀態：已確認

## 語氣與表達偏好

- 偏好白話、清楚、可直接理解的表達
- 不希望使用只有設計者自己懂的詞
- 希望概念之間邏輯清楚、層次分明

## 工作方式偏好

- 先一起討論目標，再討論做法
- 在實作前先完成較完整的規劃
- 討論定案後要同步寫入 markdown 檔案

## 禁忌與避免事項

- 避免把系統做成難以啟動的大型管理工具
- 避免把所有對話原封不動塞入知識庫
- 避免過早做過度細分或過度自動化

## 偏好候選

- 目前無
