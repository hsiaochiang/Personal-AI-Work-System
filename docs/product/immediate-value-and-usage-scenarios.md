# Immediate Value And Usage Scenarios

## 這份文件的目的

這份文件回答一個很實際的問題：

這個專案完成到某個可用階段後，能立刻帶給使用者什麼方便？第一天就能怎麼用？為什麼值得持續投入？

## 核心答案

這個專案完成後，最直接的價值不是「AI 變得很神」，而是：

- 每次開始工作時，不用重新整理背景
- 開新對話時，不用重寫一大段提示詞
- 不同對話串與不同分支之間，不容易失去脈絡
- 已經講過、改過、定過的事情，能被留下來並再次使用
- 專案做越久，AI 越能靠近使用者的工作方式

## 完成後可立即使用的方便性

### 1. 新對話啟動更快

原本情況：

- 每次開 ChatGPT、Gemini、VS Code 新對話，都要重新交代背景
- 同一個專案做久後，很容易忘記前面已經定過什麼

完成後：

- 先讀專案入口與相關記憶檔，就能快速開始
- 使用 handoff 模板，可以在短時間內建立新的規劃串、實作串、整合串

直接效益：

- 降低啟動摩擦
- 降低重複說明成本
- 降低 AI 理解偏掉的機率

### 2. 規劃與實作更容易分開管理

原本情況：

- 規劃內容、實作內容、分支內容容易混在同一串或散落在不同工具

完成後：

- 規劃有規劃檔
- 實作有 handoff
- 分支策略有明確規則
- 更新後會回寫到專案記憶

直接效益：

- 降低混亂
- 降低同一件事在不同地方說法不一致
- 更容易在多個實作串之間保持方向一致

### 3. 已定案內容不再容易遺失

原本情況：

- 很多重要決策只留在聊天裡
- 過一段時間後很難找回來

完成後：

- 決策會寫入 `decision-log.md`
- 偏好會寫入 `preference-rules.md`
- 專案背景會寫入 `project-context.md`

直接效益：

- 之後不必反覆討論同一件事
- 新對話可以直接繼承既有脈絡
- 長期工作比較不會漂移

### 4. 可逐步把高頻工作變成固定做法

原本情況：

- 每次做類似工作，都要重新想流程
- prompt 與習慣只能靠記憶

完成後：

- 重複任務會進入 `task-patterns.md`
- 穩定交付方式會進入 `output-patterns.md`
- 高頻模式會先進入 `skill-candidates.md`

直接效益：

- 每次工作更容易直接複用
- prompt 會越來越短
- 工作方式會越來越穩定

### 5. 你會有一個真正可持續成長的工作入口

原本情況：

- 工具很多，但沒有一個地方能把知識、工作流、決策與對話交接串起來

完成後：

- 這個專案本身就是工作入口
- 長期會收斂成個人 AI 工作台
- 短期先用 markdown 與 handoff 跑通流程

直接效益：

- 不用每次從零開始
- 不用再依賴零散提示詞
- 專案經驗可累積成真正的工作資產

## 完成後第一天可以怎麼用

即使還沒有完整 UI，只要目前規劃的第一階段落地，就已經可以直接使用。

### 情境 1：開始一個新規劃對話

做法：

1. 打開 `README.md`
2. 補讀 `project-context.md` 與 `decision-log.md`
3. 用 `planning-handoff-template.md` 建立新對話 handoff

你得到的方便：

- 不用重新講專案目標
- 不用重新整理已定案內容
- 新對話更容易直接進入正題

### 情境 2：開始一個新實作對話

做法：

1. 打開 `conversation-and-branch-strategy.md`
2. 判斷是否真的需要新分支
3. 用 `implementation-handoff-template.md` 建立 handoff
4. 補讀相關規則與決策檔

你得到的方便：

- 不會因為想降低幻覺就亂開分支
- 不會因為換對話就失去目標邊界
- AI 更容易集中在單一實作任務

### 情境 3：完成一輪工作後沉澱

做法：

1. 參考 `update-workflow.md`
2. 整理這輪值得保留的背景、偏好、決策與模式
3. 寫入對應檔案

你得到的方便：

- 工作成果不只留在聊天裡
- 下一次再做時，起點會更高
- 專案越做越順，而不是越做越亂

## 什麼時候會開始有明顯感受

### 很快就會有感的價值

- 開新對話比較快
- 不必一直重講背景
- 分支與對話切分比較清楚

### 中期才會變明顯的價值

- AI 越來越懂你的偏好
- 重複任務越來越能複用
- 專案記憶逐步累積成工作資產

### 長期才會完全發揮的價值

- 多專案之間開始共享成熟工作模式
- 高頻模式升級成 SKILL
- UI 工作台真正成形

## 最重要的實際改變

這個專案完成後，最實際的改變不是「多了一套文件」。

真正的改變是：

- 你開始有能力把對話變成資產
- 你開始有能力把工作習慣變成可重用流程
- 你開始有能力讓 AI 協作不是一次性，而是會累積的

## 對使用者的意義

如果這個專案做成，你得到的不是單一工具，也不是另一套 prompt 管理器。

你得到的是：

- 一套更低摩擦的 AI 協作方式
- 一個能持續累積的專案記憶系統
- 一個能讓你越做越順的個人工作基礎設施
