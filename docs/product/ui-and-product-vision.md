# UI And Product Vision

## 這份文件的目的

這份文件用來回答一個核心問題：

這個專案最後完成時，使用者實際會看到什麼、怎麼使用、為什麼值得持續投入。

## 結論

這個專案最終應該要有 UI。

理由不是因為「有 UI 看起來比較完整」，而是因為這個專案的核心目標是降低摩擦、累積記憶、支援多對話與多專案工作流。若最後只有檔案與腳本，系統雖然可運作，但很難真的成為日常工作入口。

第一階段可以先沒有完整 UI，但長期成果應該是一個清楚、低摩擦、可快速進入工作的工作台。

## 產品願景

這個專案未來應該像一個「個人 AI 工作台」。

它不是聊天機器人本身，而是使用者與多個 AI 工具合作時的記憶中樞、工作入口與交接中心。

它應該幫助使用者做到：

- 快速進入某個專案，不用重講背景
- 清楚知道目前有哪些規則、決策與工作模式
- 開新對話或新分支時，能快速產生 handoff
- 在工作後快速沉澱值得保留的內容
- 隨著使用時間增加，系統越來越懂使用者的習慣與標準

## 建議的最終 UI 想像

### 1. 首頁：Project Hub

這是進入系統後第一個畫面。

應包含：

- 專案清單
- 每個專案的目前狀態
- 最近更新時間
- 下一步建議
- 快速進入規劃、實作、整合三種模式

使用感受應該是：

- 一眼知道有哪些專案
- 一眼知道每個專案現在停在哪裡
- 不需要先打開很多資料夾才能開始工作

### 2. 專案頁：Project Workspace

點進某個專案後，會看到該專案的工作台。

應包含：

- 專案摘要
- 目前目標
- 已定案事項
- 專案記憶區
- 最近對話/最近工作紀錄
- handoff 區
- 值得沉澱的候選項目

這一頁應該像是：

- 左邊是專案記憶與結構
- 中間是目前工作焦點
- 右邊是待處理候選與最近更新

### 3. 對話交接頁：Handoff Builder

這是很重要的核心 UI，不應省略。

用途：

- 準備新規劃串
- 準備新實作串
- 準備整合串

應包含：

- handoff 類型選擇
- 自動帶入專案背景
- 可編輯的目標與邊界
- 相關檔案勾選
- 驗收標準欄位
- 產生可直接貼進新對話的 handoff 內容

這一頁的核心價值是：

- 把「開新對話」變成低摩擦、低遺漏的動作

### 4. 沉澱頁：Memory Review

工作完成後，系統需要一個地方讓使用者快速確認哪些內容要寫入專案記憶。

應包含：

- 本輪候選項目列表
- 每項候選的分類建議
- 來源摘要
- 一鍵接受 / 修改 / 忽略
- 寫入後更新到對應檔案

這一頁是把「原始對話」轉成「專案資產」的核心位置。

### 5. 規則與決策頁：Rules And Decisions

應包含：

- 使用偏好
- 任務模式
- 輸出模式
- 決策紀錄
- skill 候選

這頁不需要花俏，但要非常容易查找與掃描。

## 建議的產品氣質

這個 UI 不應該像：

- 複雜的知識管理後台
- 很重的專案管理工具
- 花很多時間維護的資料庫表單系統

它比較應該像：

- 一個很輕的工作台
- 一個對話與知識之間的轉接層
- 一個讓使用者「比較容易開始工作」的入口

## 建議的設計方向

### 氣質

- 安靜
- 清楚
- 不炫技
- 不像企業級後台那樣厚重

### 使用感

- 先看得到重點，再看得到細節
- 重視快速進入，而不是完整填表
- 任何重要操作都應該少步驟

### 視覺層次

- 首頁要有明確專案卡片或列表
- 專案頁要有清楚的區塊分工
- 規則、決策、候選項應可快速掃描

## 分階段 UI 策略

### 第 1 階段：無正式 UI

先用 markdown 檔案、固定流程、handoff 模板把工作流跑通。

目標：

- 驗證這套系統真的能降低摩擦
- 找出真正高頻的操作
- 不過早設計華而不實的畫面

### 第 2 階段：輕量 UI

建立最小可用介面，先支援：

- 專案總覽
- 專案入口
- handoff 產生
- 候選沉澱確認

### 第 3 階段：完整工作台

當流程穩定後，再做：

- 對話來源整理
- 更完整的記憶檢視
- 更方便的規則編修
- 跨專案共用能力

## 對目前專案的意義

這份願景很重要，因為它代表本專案不只是「整理 prompt 與 markdown 檔案」，而是朝向一個真正能被日常使用的個人 AI 工作台前進。

目前雖然還在規劃與骨架階段，但最終成果不應只停留在檔案系統，而應該收斂成一個可見、可用、可持續進化的工作入口。
