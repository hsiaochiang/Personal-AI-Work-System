> 版本：V5 ｜ 更新日期：2026-04-05
> 適用讀者：有技術背景、使用多種 AI 工具的個人使用者

---

## 1. 快速開始

### 1.1 環境需求
- **Node.js**: 版本建議 >= 18.x
- **作業系統**: 支援 Windows, macOS, Linux
- **瀏覽器**: 建議使用 Chrome, Edge 或 Firefox

### 1.2 啟動指令
請開啟終端機並進入專案目錄，執行以下指令：

```bash
cd web
npm start
```
啟動後，系統會自動在 http://localhost:3000 開啟管理頁面。

### 1.3 首頁概覽
<!-- SCREENSHOT: 首頁（Overview）入口，顯示左側側邊欄與主要功能標籤 -->

- **Overview (首頁)**: 專案路線圖進度、治理待辦事項摘要。
- **Extract (擷取)**: 將 AI 對話內容匯入、提取知識並寫回記憶。
- **Memory (記憶)**: 按分類瀏覽、搜尋與整理專案累積的知識。
- **Handoff (交接)**: 快速產生適合 AI 讀取的任務背景文件。
- **Decisions (決策)**: 追蹤決策歷史與規則衝突偵測。
- **Settings (設定)**: 管理 API Key 與系統偏好。

## 2. 功能模組詳解

### 2.1 Overview（專案狀態）
頁面位置：`/`

Overview 是專案的儀表板，提供以下資訊：
- **Roadmap 進度**: 根據 `docs/roadmap.md` 顯示目前階段的完成百分比與 KPI。
- **治理待辦 (Governance)**: 顯示來自治理排程的待辦清單（例如：過期記憶清理、重複條目審核）。
<!-- SCREENSHOT: Overview 頁面，顯示「治理待辦」卡片 -->

### 2.2 Handoff Builder（交接文件）
頁面位置：`/handoff`

在開啟新的 AI 對話前，建議先使用 Handoff Builder 產生背景文件。
1. **選擇類型**: 根據任務選擇「規劃」、「實作」或「整合」。
2. **編輯欄位**: 系統會自動填入目前專案的最新任務狀態。
3. **複製貼上**: 點擊「複製」按鈕，將內容直接貼給新開啟的 AI 對話，確保 AI 擁有完整的上下文。

### 2.3 知識提取與寫回（Extract）
頁面位置：`/extract`

Extract 是將散落在各工具的對話「沉澱」為專案記憶的核心入口。

#### 2.3.1 支援的工具來源與格式
| 來源工具 | 支援格式 | 操作入口 |
|----------|----------|----------|
| **ChatGPT** | 官方 JSON 匯出、分享頁 Transcript 貼上、API 自動載入 | `ChatGPT` 分頁 |
| **Gemini** | 網頁複製文字 (Transcript) | `Gemini` 分頁 |
| **Claude** | 網頁複製文字 (Transcript) | `Claude` 分頁 |
| **VS Code Copilot** | 本機 Session JSONL 檔案 | `VS Code Copilot` 分頁 |
| **手動輸入** | 純文字貼上 | `純文字` 分頁 |

#### 2.3.2 操作流程
1. **選擇來源**: 在下拉選單中選擇對應的 AI 工具。
2. **匯入內容**: 貼上文字或上傳檔案。
   - **ChatGPT API**: 需先在 Settings 設定 OpenAI API Key，即可從 tracked 列表選擇最近對話。
   - **Copilot**: 直接刷新清單並點選最近的本機 Session。
3. **提取候選**: 點擊「提取候選知識」，系統會識別「背景、偏好、輸出模式、決策」四類知識。
4. **審核與寫回**: 審核每條候選（可編輯內容），點擊「寫回」。
   - **安全保護**: 每次寫回前系統會自動備份 `docs/memory/` 到 `.backup/`。

### 2.4 專案記憶（Memory）
頁面位置：`/memory`

#### 2.4.1 這一頁的目的——你來這裡要做什麼？

Memory 頁面是你**檢視並維護 AI 對你的了解程度**的地方。

系統透過 Extract（擷取）頁面從你的 AI 對話中不斷累積知識，但這些知識需要定期被審視：
- 有些已過時（例如：當初記錄「尚未有 UI」，但現在介面早已上線）
- 有些重複（從不同對話寫入了相似的規則）
- 有些來源不明（沒有 `<!-- source: -->` 標記，不知道是誰寫入的）

**你造訪這頁面的典型任務：**

| 情境 | 你應該做的操作 |
|------|--------------|
| 剛完成一個新功能，想確認記憶反映最新狀況 | 翻閱對應分類（如 `project-context.md`），手動確認描述是否仍準確 |
| 發現某個分類的「待確認」比例很高 | 點開 Health Overview → 找出具體條目 → 確認後手動更新或刪除 |
| 懷疑有些條目已重複 | 查看 Dedup Overview 的建議，點選合併 |
| 想讓 AI 對你有更好的理解 | 先去 Extract 擷取最近幾次有產出的對話，再回這裡確認寫入正確 |
| 想知道記憶整體品質如何 | 點擊右上角的「AI 品質審查」，由 Gemini 分析所有記憶檔案並給出改善建議 |

#### 2.4.2 頁面結構解說

**KPI 卡片（頂部四格）**

| 指標 | 說明 | 健康目標 |
|------|------|---------|
| 記憶條目 | 所有分類合計的條目數 | 視規模，50~200 條屬正常 |
| 檔案數 | 目前存在的記憶檔案數量 | 通常為 6 個 |
| 過期比例 | 超過 60 天未更新的條目比例 | 低於 30% |
| 建議清理 | 系統建議可刪除或合併的條目數 | 趨近 0 |

**Health Overview（健康度總覽）**

每個記憶檔案的整體健康狀況：
- 🟢 **健康**：條目有明確日期與來源，近期更新過
- 🟡 **待確認**：缺乏 `<!-- source: ... -->` 標記，或超過 30 天未更新
- 🔴 **過期風險**：超過 60 天未更新，內容可能已失效

> 行動建議：若某檔案全為「待確認」，請直接打開 `docs/memory/<檔案名>.md` 在編輯器中手動增補來源標記與日期。

**Dedup Overview（去重建議）**

當系統偵測到兩條以上語意相近的條目，會在此標記。操作方式：
1. 點擊建議項目展開比較
2. 選擇「合併」（保留較完整的，刪除較舊的）或「保留」（兩條都需要）

**記憶詳細內容（下半部分類卡片）**

展開每個分類可查看個別條目。你可以直接在此：
- 確認條目內容是否仍正確
- 點擊「刪除」移除已不相關的條目

#### 2.4.3 各記憶分類說明

| 檔案名稱 | 儲存內容 | 典型更新時機 |
|---------|---------|------------|
| `decision-log.md` | 技術決策記錄（選哪個模型、為何修改某架構）| 每次做出重要技術決策後 |
| `preference-rules.md` | 你的工作偏好（例如：「回覆要簡短」、「優先用正體中文」）| 首次設定後通常穩定 |
| `output-patterns.md` | AI 輸出品質的偏好（風格、格式）| 有明顯輸出偏好改變時 |
| `project-context.md` | 專案背景與功能邊界（現在做到哪、不做什麼）| 每個版本里程碑結束後 |
| `skill-candidates.md` | 值得固化為 SKILL 的重複操作模式 | 累積 3 次以上相同操作後 |
| `task-patterns.md` | 任務類型與處理模式（如「debug 流程」）| AI 輔助提取後人工確認 |

#### 2.4.4 建議的日常維護流程

```
每週 5 分鐘例行檢查：
1. 開啟 /memory
2. 查看 KPI：過期比例是否上升？建議清理項目多嗎？
3. 若有 🔴 條目 → 確認是否仍適用，不適用則刪除
4. 若有去重建議 → 合併相近條目
5. 點「AI 品質審查」→ 查看 Gemini 給出的改善建議
6. 根據建議，手動更新 1-2 條記憶

每個功能版本結束後：
1. 更新 project-context.md（現在做到哪）
2. 更新 decision-log.md（本版本做了哪些重要決策）
3. 將 skill-candidates.md 中「已實現」的候選升級為「已確認」
```

### 2.5 決策與規則檢視（Decisions）
頁面位置：`/decisions`

- **決策記錄**: 追蹤所有定案的技術決策。
- **規則與衝突**: 顯示所有工作模式與偏好。若有兩條規則互相矛盾（例如：一條要求「簡短」，一條要求「詳細」），系統會高亮標記衝突原因。

### 2.6 設定管理（Settings）
頁面位置：`/settings`

#### 2.6.1 OpenAI API Key 設定
若要啟用 ChatGPT API 自動載入功能，請在此輸入您的 OpenAI API Key。
- **安全性**: 密鑰僅儲存在伺服器端 `web/api-keys.json`。系統**不會**將您的 Key 上傳到任何外部服務，且該設定檔已列入 `.gitignore` 以防不慎 commit。

## 3. 多工具整合指南 (V5)

V5 版本大幅強化了多工具的整合能力，以下是各工具的最佳實踐：

### 3.1 ChatGPT 整合
- **API 載入 (推薦)**: 在 Settings 填入 Key 後，可追蹤 `conversationId` 進行自動同步，省去頻繁匯出。
- **JSON 匯出**: 從 ChatGPT 網頁點選「Settings」→「Data controls」→「Export data」，上傳匯出的 `conversations.json`。

### 3.2 Gemini 與 Claude 整合
- **半自動貼上**: 開啟 Gemini 或 Claude 對話，直接全選複製 (Ctrl+A / Ctrl+C)，在 Extract 頁面選對來源並貼上即可。系統會自動過濾 `You` / `Gemini` / `Human` / `Claude` 等標記並解析對話輪次。

### 3.3 VS Code Copilot 整合
- **本機 Session**: 系統會掃描 VS Code 的 global storage 路徑。直接點擊「刷新」，從清單中找出您最近在 VS Code 中進行的對話即可載入。

## 4. 常見問題與已知限制

### 4.1 常見問題
- **找不到 API Key 設定處？**: 請確認已開啟 http://localhost:3000/settings。
- **貼上文字後偵測不到候選？**: 請確認貼上的文字包含完整的對話脈絡（包含使用者問題與 AI 回答）。
- **想清理舊記憶？**: 前往 `/memory` 頁面，參考「健康度」標記進行人工整理。

### 4.2 已知限制 (V5)
- **API 自動載入**: 僅支援 OpenAI (ChatGPT)。Gemini 與 Claude 目前不提供官方對話歷史 API，故僅支援貼上 Transcript。
- **多使用者**: 本系統設計為個人離線工作台，目前不支援多帳號或協作。
- **自動執行**: 治理排程僅提供「建議」，不論是記憶清理還是規則合併，最後一步都必須經過人工點擊確認。

### 4.3 資料安全與備份
- 所有知識資料均以 Markdown 格式存儲在 `docs/` 資料夾下。
- **強烈建議**: 定期將整個專案目錄透過 Git 進行備份。
- 密鑰儲存於 `web/api-keys.json`，請勿手動移除此檔案的 `.gitignore` 排除規則。
