# V1 Roadmap — 當前衝刺目標

> V1 = 這套系統能在日常工作中真正被使用的最小版本。  
> 完整專案路線請看 [`project-roadmap.md`](project-roadmap.md)。

> 狀態更新：2026-03-26（對齊來源：`docs/roadmap.md`、`docs/handoff/current-task.md`）

## V1 完成定義

V1 完成時，應至少具備：

- [x] 穩定的專案記憶結構
- [x] 穩定的 handoff 工作流
- [x] 可執行的對話紀錄提取流程
- [x] 可人工確認後回寫的沉澱流程
- [x] 至少一條「對話紀錄 → 候選內容 → 更新專案記憶」的閉環
- [x] 使用者在新對話中實際感受到啟動成本下降

## 前提假設

- 一位主要使用者 + AI 協作
- 各階段可連續推進
- 不先做正式 UI、不先做多工具同步
- 先追求可用，再追求自動化與完整性

## V1 邊界

| 做 | 不做 |
|---|-----|
| 專案記憶檔案 | 正式 UI |
| handoff 模板 | 多工具同步 |
| 更新流程 | 完整 personal/shared 層 |
| VS Code 對話紀錄提取 MVP | 向量資料庫 |
| 半自動沉澱流程 | 自動提取（無人工確認） |
| 真實專案驗證 | 多專案模板管理 |

V1 不應做：

- 正式 UI
- 多工具同步
- 自動建立 SKILL
- 向量資料庫
- 複雜搜尋
- 個人層與共享層的完整系統化

## 建議時程版本

### 版本 A：2 週衝刺版

適合：

- 想盡快得到可用版本
- 接受先有些粗糙，但先能用

安排：

- 第 1 週：Phase 1 + Phase 2 前半
- 第 2 週：Phase 2 後半 + Phase 3 縮小版 + Phase 4

風險：

- 可能會留下較多後續修正項

### 版本 B：4 週穩健版

適合：

- 想讓第一版一開始就更穩
- 願意多花時間做真實驗證

安排：

- 第 1 週：Phase 1
- 第 2 週：Phase 2
- 第 3 週：Phase 3
- 第 4 週：Phase 4

優點：

- 更有機會做出真的能持續使用的版本

## 我對目前專案的建議

如果以目前狀態來看，我建議採：

- **4 週穩健版**

原因：

- 這個專案的核心價值在於工作流與沉澱品質，不在於趕快堆功能
- 若太快跳到自動化，容易做出能跑但不好用的系統
- 先把「真的用得起來」做穩，後面再擴充 UI 與多工具能力會更順
