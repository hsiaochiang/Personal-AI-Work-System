# V1 Roadmap

## 目的

定義本專案第一版從現在到可用狀態的實際推進路線，包含：

- 第一版要完成哪些內容
- 每個階段要做哪些工作
- 預估需要多少時間
- 每個階段完成後可以立刻怎麼使用

## V1 的完成定義

第一版完成，不代表整個產品做完，而是代表這套系統已經能在日常工作中真正使用。

V1 完成時，應至少具備：

- 穩定的專案記憶結構
- 穩定的 handoff 工作流
- 可執行的對話紀錄提取流程
- 可人工確認後回寫的沉澱流程
- 至少一條從「對話紀錄 -> 候選內容 -> 更新專案記憶」的閉環
- 使用者能在新對話中實際感受到啟動成本下降

## 時間估算前提

以下估算以「一位主要使用者 + AI 協作」為前提，且假設：

- 不同階段之間可連續推進
- 不先做正式 UI
- 不先做多工具同步
- 先追求可用，再追求自動化與完整性

估算採保守區間，不以最理想情況計算。

## 總時程概覽

### 快速版

- 約 2 到 3 週可完成 V1 最小可用版本

### 穩健版

- 約 4 到 6 週可完成較穩定的 V1

差異主要取決於：

- 對話紀錄提取流程是否順利
- 手動審核與回寫規則是否需要多次修正
- 是否會在中途擴大範圍

## 階段拆解

## Phase 0：規劃與骨架建立

### 目標

- 完成規劃
- 建立專案記憶骨架
- 建立 handoff 與更新流程

### 主要工作項目

- 定義目標、原則、策略、運作方式
- 建立專案記憶檔案結構
- 建立 handoff 模板
- 建立更新流程
- 定義 UI 願景與資訊架構

### 目前狀態

- 已完成

### 完成後可立即使用

- 已可用 markdown 檔案與 handoff 模板開始手動工作

## Phase 1：手動流程跑通

### 目標

- 不靠程式，自行把這套流程完整跑一次到多次
- 驗證檔案結構與工作流真的可用

### 主要工作項目

- 用現有模板實際開至少 2 到 3 次新對話
- 用 handoff 模板交接規劃串與實作串
- 在每次工作後手動更新專案記憶檔
- 觀察哪些欄位好用、哪些欄位太多、哪些檔案其實沒被看
- 修正模板與更新流程

### 交付成果

- 修正版模板檔
- 修正版更新流程
- 至少一批真實沉澱內容

### 預估時間

- 3 到 5 天

### 完成後可立即使用

- 即使完全沒有自動化，也能開始穩定用這套系統管理單一專案

## Phase 2：半自動提取 MVP

### 目標

- 建立第一版半自動提取流程
- 讓對話紀錄可以轉成候選沉澱項目

### 主要工作項目

- 研究 VS Code / Copilot Chat 對話紀錄格式
- 定義提取輸入與輸出格式
- 設計候選項目的最小 schema
- 建立第一版提取腳本或提取流程
- 讓提取結果能對應到：
  - `project-context.md`
  - `preference-rules.md`
  - `task-patterns.md`
  - `decision-log.md`
- 建立人工確認後寫回檔案的流程

### 交付成果

- 第一版提取流程
- 第一版候選輸出格式
- 第一版回寫規則

### 預估時間

- 5 到 8 天

### 完成後可立即使用

- 可以從真實對話中半自動產生沉澱候選，明顯降低手動整理成本

## Phase 3：真實專案驗證

### 目標

- 在真實專案裡驗證這套系統是否真的降低摩擦
- 找出第一版最需要修的地方

### 主要工作項目

- 用至少 1 個真實專案跑完整流程
- 觀察：
  - 新對話啟動是否更快
  - 是否更少重講背景
  - AI 首輪回覆是否更貼近需求
  - 決策是否真的不再遺失
- 收集實際使用時的問題
- 修正：
  - 檔案結構
  - 提取規則
  - handoff 模板
  - 更新流程

### 交付成果

- 真實專案驗證紀錄
- 修正版規則與模板
- 更穩定的 V1 工作流

### 預估時間

- 1 到 2 週

### 完成後可立即使用

- 可以把這套系統當成真正的日常工作入口，而不只是實驗

## Phase 4：V1 收斂與完成定版

### 目標

- 把已驗證有效的部分整理成穩定版本
- 明確界定 V1 的完成邊界

### 主要工作項目

- 收斂最終 V1 檔案結構
- 收斂最終提取分類與沉澱規則
- 收斂 handoff 與分支使用策略
- 定義哪些內容進 V2，不再放進 V1
- 更新：
  - `README.md`
  - `decision-log.md`
  - `ai-workflow-planning.md`
- 製作一份 V1 使用說明

### 交付成果

- V1 定版文件
- 可持續使用的第一版工作系統

### 預估時間

- 2 到 4 天

### 完成後可立即使用

- 可作為其他專案的起始模板與工作基礎設施

## V1 工作項目總表

以下是 V1 預計要完成的主要工作項目：

### A. 規劃與結構

- 完成規劃文件
- 完成專案記憶結構
- 完成更新流程
- 完成對話與分支策略
- 完成 handoff 模板

### B. 使用與驗證

- 至少跑 2 到 3 次手動工作流程
- 至少跑 1 個真實專案驗證
- 記錄使用中出現的摩擦點

### C. 提取與沉澱

- 定義對話紀錄輸入格式
- 定義候選輸出格式
- 建立第一版提取流程
- 建立人工確認後回寫流程

### D. 收斂與完成

- 修正模板
- 修正分類規則
- 修正更新流程
- 定義 V1 完成邊界

## 建議的 V1 邊界

V1 應做：

- 專案記憶檔案
- handoff 模板
- 更新流程
- VS Code 對話紀錄提取 MVP
- 半自動沉澱流程
- 真實專案驗證

V1 不應做：

- 正式 UI
- 多工具同步
- 自動建立 SKILL
- 向量資料庫
- 複雜搜尋
- 個人層與共享層的完整系統化

## 建議時程版本

### 版本 A：2 週衝刺版

適合：

- 想盡快得到可用版本
- 接受先有些粗糙，但先能用

安排：

- 第 1 週：Phase 1 + Phase 2 前半
- 第 2 週：Phase 2 後半 + Phase 3 縮小版 + Phase 4

風險：

- 可能會留下較多後續修正項

### 版本 B：4 週穩健版

適合：

- 想讓第一版一開始就更穩
- 願意多花時間做真實驗證

安排：

- 第 1 週：Phase 1
- 第 2 週：Phase 2
- 第 3 週：Phase 3
- 第 4 週：Phase 4

優點：

- 更有機會做出真的能持續使用的版本

## 我對目前專案的建議

如果以目前狀態來看，我建議採：

- **4 週穩健版**

原因：

- 這個專案的核心價值在於工作流與沉澱品質，不在於趕快堆功能
- 若太快跳到自動化，容易做出能跑但不好用的系統
- 先把「真的用得起來」做穩，後面再擴充 UI 與多工具能力會更順

## 下一步

進入 V1 實作前，最合理的下一步是：

1. 把 Phase 1 拆成具體任務清單
2. 選一個真實專案作為第一個驗證專案
3. 定義對話紀錄提取 MVP 的輸入、輸出與人工確認方式
