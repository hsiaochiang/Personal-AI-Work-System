# Project Roadmap

## 目的

這份文件描述整個專案從現在到長期目標的演進路線。

`v1-roadmap.md` 著重第一版可用版本。  
這份文件則回答更大的問題：

- 這個專案後面會長成什麼
- 各版本的邊界在哪裡
- 每一階段完成後會增加哪些能力
- 整體大約需要多久

## 全專案願景

這個專案最終要從一套 markdown 驅動的工作流，成長為一個真正可用的「個人 AI 工作台」。

它會逐步具備：

- 專案記憶
- 個人工作偏好
- 對話交接
- 對話紀錄提取
- 工作流沉澱
- 可重用的 SKILL
- 輕量 UI
- 多工具接入能力

## 階段總覽

## Phase 0：規劃與骨架

### 目標

- 定義方向
- 建立文件結構
- 建立初始工作流骨架

### 狀態

- 已完成

### 產出

- 規劃文件
- 專案記憶檔
- handoff 模板
- 更新流程
- UI 願景與資訊架構

## Phase 1：V1 可用工作流

### 目標

- 先讓這套系統在沒有正式 UI 的情況下就能真正被使用

### 主要能力

- 專案記憶可用
- handoff 可用
- 手動流程可用
- VS Code 對話紀錄提取 MVP
- 半自動沉澱流程
- 真實專案驗證

### 邊界

- 不做正式 UI
- 不做多工具同步
- 不做完整 personal/shared 層

### 預估時間

- 4 週穩健版
- 最快約 2 到 3 週可做出最小可用版本

### 完成後價值

- 可以立刻成為單一專案的 AI 工作入口
- 可明顯降低開新對話的摩擦
- 可開始把工作成果轉成長期資產

## Phase 2：V1.5 穩定化與產品化前準備

### 目標

- 把 V1 從「可用」提升到「穩定、可重複、可轉移」

### 主要能力

- 更穩定的提取規則
- 更清楚的候選審核流程
- 更成熟的記憶檔模板
- 更清楚的第一版使用說明
- 可重用於第二個、第三個專案

### 可能工作項目

- 提取流程的重構
- 模板欄位收斂
- 對真實案例做回顧
- 新專案初始化流程
- 專案層與個人層的邊界定義

### 預估時間

- 2 到 4 週

### 完成後價值

- 第一版不再只是這一個專案專用
- 可以更穩定地複製到其他專案使用

## Phase 3：V2 輕量 UI 工作台

### 目標

- 把高頻操作從純 markdown 工作流提升為輕量 UI

### 主要能力

- Projects
- Workspace
- Handoffs
- Memory Review

### 可能工作項目

- UI MVP 設計與實作
- 以現有 markdown 為資料來源
- 專案總覽與專案頁
- handoff builder
- 候選審核介面

### 預估時間

- 4 到 8 週

### 完成後價值

- 日常使用摩擦再下降
- 工作入口更直覺
- 不需先翻檔案也能開始工作

## Phase 4：V2.5 多專案與共享能力

### 目標

- 從單一專案記憶，進一步發展出跨專案可重用能力

### 主要能力

- `personal/` 層
- `shared/` 層
- 跨專案共用規則
- 更成熟的 SKILL 候選管理

### 可能工作項目

- 個人偏好與專案偏好的正式分層
- shared workflow 整理
- 跨專案模板
- 技能候選升級流程

### 預估時間

- 3 到 6 週

### 完成後價值

- 新專案啟動速度明顯提升
- 多個專案開始共用成熟工作方式

## Phase 5：V3 多工具接入與完整個人工作台

### 目標

- 讓這套系統從單一工具來源，成長為真正跨工具的工作台

### 主要能力

- ChatGPT 對話接入
- Gemini 對話接入
- Antigravity 對話接入
- 多來源記憶整合
- 更成熟的記憶檢索與工作建議

### 可能工作項目

- 多工具輸入格式整合
- 統一提取與沉澱層
- 更完善的審核與回寫流程
- 進一步的 UI 擴充

### 預估時間

- 6 到 10 週

### 完成後價值

- 這套系統會真正成為跨工具的個人 AI 工作基礎設施

## 總時程估算

### 保守估算

- 從現在到 V1 完成：約 4 週
- 到 V2 輕量 UI：約 2 到 4 個月
- 到 V3 多工具工作台：約 4 到 6 個月以上

### 重要說明

這不代表一定要把整條路走完才有價值。

事實上：

- 到 V1 就已經有很高的立即使用價值
- 到 V2 才會開始有更明顯的產品感
- 到 V3 才接近完整願景

## 建議推進方式

### 目前建議

- 先專注完成 V1
- 暫時不要提前做 V2 的正式 UI
- 暫時不要提前做 V3 的多工具整合

### 原因

- 本專案目前最需要的是驗證工作流，而不是擴大範圍
- 若 V1 沒有真的可用，後面的 UI 與整合都會建立在不穩的基礎上

## 對使用者的意義

這份 roadmap 的重點不是把事情排得很遠，而是讓你知道：

- 現在做的每一步不是零散的
- 目前的 markdown、handoff、工作流不是過渡垃圾，而是未來產品的核心骨架
- 後續 UI 與多工具能力會建立在現在這套結構上，而不是推翻重來
