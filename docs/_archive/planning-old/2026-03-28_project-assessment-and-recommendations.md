# 2026-03-28 專案重新審視與建議

## 目的

重新檢視目前專案的實作進度、現行 roadmap 與原始目標之間的對齊程度，回答：

- 專案目前做到哪裡
- 目前規劃的下一步能不能達到原本想要的目標
- 現階段已做到哪些規劃中的功能
- 接下來應該如何調整，才能更接近真正想要的成效

## 快速結論

### 結論 1：專案已經從純文件規劃進入可操作產品雛形

目前不再只是規劃文件，而是已有可啟動的本地網站、基本資料讀取 API 與三個可瀏覽頁面：

- Overview
- Current Task
- Memory

這代表專案已進入開發階段，且 Phase 1 的最小可用成果已經存在。

### 結論 2：目前做出的東西，還沒有直接達成原始核心目標

原始核心目標是：

- 建立核心知識庫
- 讓專案記憶持續累積
- 讓 AI 越來越懂使用者需求
- 降低新對話與新專案啟動摩擦

但目前真正已實作的內容，主要是：

- 讀取 markdown
- 顯示專案狀態
- 顯示當前任務
- 顯示 memory 檔內容

這些功能有幫助，但它們比較接近「可視化工作台的第一層」，還沒有形成原始目標最重要的能力：

- 對話紀錄提取
- 候選知識審核
- 寫回專案記憶
- 新對話的低摩擦交接

### 結論 3：如果完全照目前 roadmap 順序往下做，可能無法最快達到最想要的成效

原因是：

- 現在 roadmap 把「Extraction Engine」放在最後一個主要 Phase
- 但原始目標裡最核心的價值，正是「把對話變成資產」
- 若先投入太多時間做 UI 完整度，而把提取與沉澱能力延後，產品會先變得更好看，但不一定更有用

所以：

現行 roadmap **不是錯的**，但**優先順序需要調整**。

而這其實是最直接降低新對話摩擦的功能之一。

### C. 決策與規則雖有文件，但尚未形成有效工作流

目前：

但還沒有：

這不是錯，但需要明確區分：

- 目前可用
- V1 完成後可用

否則容易讓人高估現在的實際能力。

## 現行 roadmap 的主要風險

## 風險 1：先做 UI 完整度，後做核心價值

目前 roadmap 把：

- 頂部導覽
- 側邊欄
- 搜尋
- Detail 頁
- 多專案

排在提取引擎之前。

問題是：

使用者最初真正想解的是「知識累積與啟動摩擦」，不是「先有更完整的 dashboard」。

如果先做大量 UI，會有兩個風險：

- 做出一個很好看的檢視器，但還不能真正讓 AI 變聰明
- 產品看起來進展很多，但使用者最在意的核心價值還沒有實現

## 風險 2：V1 定義過大

目前 `docs/roadmap.md` 把 Phase 1–5 全部視為 V1 完整產品。

這種定義有一個問題：

- V1 太大，會讓「完成」變得遙遠
- 使用者已經有可用成果，但心理上仍覺得離完成很遠

這不利於建立節奏感與成就感。

## 風險 3：產品方向可能偏成「專案 dashboard」

目前實作很自然地朝 dashboard 發展，這是合理的第一步。

但若不持續提醒核心目標，產品會慢慢偏成：

- 讀文件
- 看狀態
- 看任務

而不是：

- 累積知識
- 幫助交接
- 讓 AI 協作更有效

## 我對目前狀態的判斷

### 是否有做到規劃出來的功能？

有做到一部分，而且是合理的一部分。

已做到的是：

- Phase 1 入口型功能
- 文件到頁面的第一層映射
- 本地可運行工作台骨架

但還沒有做到最關鍵的中樞功能。

### 是否能達到原本目標？

可以，但前提是後續執行順序要調整。

如果後面幾個階段都順著「先做更完整 dashboard，再做提取引擎」走，會比較慢碰到真正高價值。

如果把提取與審核閉環提前，則很有機會達到原本目標。

## 建議調整方向

## 建議 1：把「提取 + 審核 + 寫回」提前到下一個主要里程碑

我建議把原本 roadmap 的後半核心能力提前，至少做出最小閉環：

1. 貼上對話
2. 產生候選
3. 人工確認
4. 寫回 `docs/memory/*.md`

這個功能一旦成立，才真正開始兌現原始承諾。

### 調整理由

- 這是原始目標的核心
- 這比搜尋、Detail 頁、KPI 更能直接帶來價值
- 做完之後，AI 協作會有明顯「越用越懂」的感覺

## 建議 2：把 handoff builder 提前，不要再延後太久

handoff 不是附屬功能，而是這個產品能不能真的降低新對話摩擦的核心之一。

建議至少先做最小版本：

- 選類型
- 帶入專案背景
- 複製可貼上的 handoff

不一定要完整頁面很漂亮，但功能應該提早存在。

## 建議 3：把 roadmap 的里程碑改成「價值導向」而不是「頁面導向」

目前 roadmap 比較像：

- 先做哪些頁面
- 再做哪些頁面

我建議改成：

- 先解決哪個核心價值
- 再做支撐這個價值的頁面

建議順序：

1. 入口與基本檢視
2. handoff 降摩擦
3. 對話提取與候選審核
4. 記憶寫回與決策追蹤
5. 多專案與完整 UI

## 建議 4：把 `user-guide-v1.md` 分成「現在可用」與「V1 完成後」

目前使用手冊描述的很多情境，是完成後願景，不是現在就能做到。

建議拆成兩層：

- Current User Guide
- V1 Final User Guide

這樣使用者對現況與未來不會混淆。

## 建議 5：建立「是否真的更接近目標」的量化檢查點

目前 roadmap 偏向功能驗收，但還不夠檢驗原始目標是否真的達成。

建議增加這些檢查點：

- 新對話平均需要補充多少背景
- handoff 是否能讓新對話更快進入工作
- 決策是否能被重用，而不是重講
- 對話提取後，是否真的寫回有效記憶
- 下次 AI 回覆是否更貼近既有偏好

## 建議的下一步執行順序

若從現在開始調整，我建議：

### Step 1

先完成 Phase 2 中真正必要的 UI 重構項，但不要把所有 polish 當成優先：

- 保留基本導覽
- 保留 Detail 頁必要資訊
- 不把視覺完整度放在最前面

### Step 2

提前做最小 handoff builder

### Step 3

提前做最小 extraction + candidate review + writeback

### Step 4

等核心閉環可用後，再補完整搜尋、多專案、更多頁面細節

## 我最核心的判斷

這個專案目前方向沒有偏掉，但**重心有偏向 UI 工作台，而不是先把知識閉環做出來**。

如果現在就調整，還完全來得及，而且調整成本不高。

最重要的是不要把：

- 好看的儀表板

誤當成：

- 已經完成了你原本真正想要的個人 AI 工作系統

真正讓這個專案產生長期價值的，不是 dashboard 本身，而是：

- 對話能不能變成資產
- 資產能不能被下次工作重用
- 新對話能不能更快、更準地接上原本脈絡

## 建議總結

### 我建議保留

- 現在的網站方向
- 現在的文件結構
- 現在的 roadmap 作為產品化基礎

### 我建議調整

- 把 extraction / review / writeback 提前
- 把 handoff builder 提前
- 把 UI polish 放到核心價值之後
- 把 user guide 拆成 current / final 兩層
- 把驗收加上「是否真的降低摩擦」的衡量

## 最後結論

這個專案**有潛力達到原本規劃的目標**，而且目前並沒有走到錯方向。

但如果不調整優先順序，產品可能會先長成一個好看的工作台，而不是一個真正會讓 AI 協作越用越有效的系統。

現在最值得做的事不是推翻，而是校準。
