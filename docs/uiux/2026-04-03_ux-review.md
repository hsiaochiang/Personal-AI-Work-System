# UX Review 2026-04-03

## Review Scope

- 驗收 `chatgpt-adapter` 對 `/extract` 使用流程的最小 UX 變更
- 檢查重點：
  - 既有 textarea 仍是唯一主輸入面
  - 上傳 JSON/TXT 後不需要跳到新的 flow 或頁面
  - plain text fallback 仍可直接使用

## Findings

- `/extract` 沒有被擴大成多來源 selector；使用者仍只面對單一輸入區與單一「提取候選知識」主按鈕。
- 檔案上傳只是把內容載入既有 textarea，沒有新增第二條 extraction path，降低使用者分流成本。
- 使用者在檔案載入後若直接修改 textarea，狀態會切回手動貼上模式，避免沿用過期的 JSON hint。
- ChatGPT JSON 多 conversation 的限制目前仍存在，但已用 deterministic 規則與文案說明控制風險；完整挑選留給後續 change。

## Decision

- UX review PASS
- 目前流程維持單一路徑與低摩擦，符合本 change 的最小安全範圍

## Evidence

- `web/public/extract.html`
- `web/public/js/extract.js`
- `docs/qa/2026-04-03_chatgpt-adapter-smoke.md`

## Addendum — local-import-vscode-copilot

### Review Scope

- 驗收 `local-import-vscode-copilot` 對 `/extract` 操作流程的最小 UX 變更
- 檢查重點：
  - 使用者是否能在同一頁完成「刷新 session → 載入 → 提取候選」
  - 路徑錯誤、空清單、載入成功與手動編輯切回等狀態是否明確
  - 既有 ChatGPT / plain text 流程是否仍可共存

### Findings

- 使用者可在同一張卡片內完成 path override、session 刷新、單筆載入與既有 extraction，沒有被迫切換到第二條流程。
- 載入 Copilot session 後，textarea 會顯示 preview；若使用者手動編輯，系統會切回一般 auto-detect 模式，避免來源混淆。
- 清單刷新失敗時會透過 status 顯示錯誤，空清單也有單獨 empty state，不會讓使用者誤判為畫面卡住。
- ChatGPT 檔案上傳與一般貼上入口仍保留在同頁，確保多來源整合前的過渡 UX 可用。

### Decision

- UX review PASS
- 本次流程維持低摩擦、狀態清楚，符合本 change 的 MVP 目標

- `web/public/extract.html`
