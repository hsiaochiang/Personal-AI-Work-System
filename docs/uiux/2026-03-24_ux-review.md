# UX Review 2026-03-24

## Flow

- Flow 名稱：第一次 OpenSpec 手動 workflow pilot
- 入口：`AGENTS.md` → `docs/handoff/current-task.md` → `docs/roadmap.md` → `docs/agents/commands.md`
- 主要步驟：
  1. 確認目前 task、scope 與 blockers
  2. 建立 active OpenSpec change
  3. 用 instructions 補齊 artifacts
  4. 驗證 change 與 repo smoke
  5. 寫回 QA、runlog、memory、handoff

## States

- Loading: 無系統 loading UI；以 runlog 記錄進行中的步驟
- Success: strict validate 與 smoke 通過，且 handoff 能交給下一位 agent
- Error: validate 失敗或 repo smoke 失敗時，需把錯誤寫回 QA / handoff，而不是只留在對話
- Exit condition: 若剩餘步驟涉及 commit / push / archive，需等待人工確認

## Findings

- 第一次 pilot 的最大 UX friction 在於：不知道先跑 `openspec new change`，就無法從 CLI 取得 artifact 格式說明
- handoff 與 QA 組合後，已足以讓下一位 agent 接手；其中 QA 更適合承載摩擦紀錄，handoff 則應維持短而可續接

## DoD

- [x] 有完整入口與下一步
- [x] 有成功與錯誤狀態處理
- [x] 有可重播的收尾證據
- [x] 明確標示不可逆操作需人工確認

## Flow（Pilot 2）

- Flow 名稱：第 2 次 OpenSpec 手動 workflow pilot 比較驗證
- 入口：`AGENTS.md` → `docs/handoff/current-task.md` → `docs/handoff/blockers.md` → `docs/roadmap.md` → `docs/agents/commands.md`
- 主要步驟：
  1. 依固定入口順序判定目前狀態與下一步
  2. 以 `openspec instructions` 直接取得 proposal、specs、design、tasks 格式
  3. 以 `openspec change validate phase1-entrypoint-guidance-pilot --strict` 完成 strict validate
  4. 執行 workspace `--verify-only` smoke
  5. 寫回 QA、tasks、runlog、handoff、roadmap 與 review 證據

## States（Pilot 2）

- Loading: 以 runlog 記錄第 2 次 pilot 的實際入口順序與命令
- Success: `openspec instructions`、strict validate 與 smoke 均成功，且 QA 完成與第 1 次 pilot 的直接比較
- Error: 若 instructions、validate 或 smoke 任一失敗，需把偏差與阻塞寫回 QA / handoff，而不是只留在對話
- Exit condition: 本次依要求停在 commit 前；後續 commit / sync / archive 需等人工確認

## Findings（Pilot 2）

- 相較第 1 次 pilot，artifact 格式與 strict validate 入口都已改為直接命中，沒有再靠 help 試錯或 validate 命令反覆修正
- 額外文件閱讀仍存在，但已從「為了找入口而掃描」收斂成「在固定入口後，為了比較與驗證而定向補讀」

## DoD（Pilot 2）

- [x] 固定入口順序已實際重播
- [x] `openspec instructions` 的使用已留下命令級證據
- [x] `openspec change validate ... --strict` 已作為唯一 strict validate 入口
- [x] 已明確寫出與第 1 次 pilot 的比較結論
- [x] 已標示 commit / sync / archive 仍需人工確認
