# 個人 AI 工作系統規劃

## 目前已定版的目標

- 建立一套可跨 ChatGPT、Gemini、VS Code、Antigravity 使用的核心知識庫
- 讓每個專案都能逐步累積專屬記憶，而不是只累積聊天紀錄
- 讓重複任務逐步從自由提示詞，收斂成固定工作流與 SKILL
- 讓 AI 在同一專案中越來越懂我的標準，並降低每次啟動摩擦
- 讓新專案可以從既有模板快速開始，而不是重頭建制

## 問題重述

目前遇到的核心問題不是單純的提示詞管理，而是缺少一套可持續累積、可跨工具重用、可逐步降低摩擦的工作記憶系統。

既有狀況：

- 已投入大量時間建立提示詞管理系統，但實際上沒有順利進入日常工作流
- 提示詞分散在不同工具中，難以共享與演化
- 真正需要累積的是偏好、規格、決策脈絡、任務模式與工作習慣，而不只是 prompt 本身

## 規劃方法論

目前採用的規劃順序：

1. 目標
2. 原則
3. 策略
4. 運作方式
5. MVP
6. 回顧迭代

這個順序的目的，是避免過早跳入工具、資料夾結構或自動化實作，導致做到一半才發現方向不合適。

## 目前已確認的方向

- 對話紀錄是很有價值的原始材料
- 不應直接把原始對話當成知識庫使用
- 應建立一層「提取與沉澱」機制，把對話轉成可重用的資產
- 第一階段應優先處理最容易取得、格式最清楚的資料來源
- 初期較適合採用半自動模式，而不是全自動模式

## 設計原則

1. 低摩擦優先
   系統必須容易啟動、容易使用。只要某個流程明顯增加啟動成本，就算設計上很完整，也不適合作為核心做法。

2. 知識資產優先於提示詞資產
   真正要累積的是偏好、規格、決策、術語、範例與流程，不是大量 prompt。prompt 只保留啟動任務需要的最小內容。

3. 半自動優先於全自動
   初期由系統自動提議、人工確認後再沉澱，比完全自動寫入更可靠。先追求可控，再追求完全自動。

4. 專案記憶與個人記憶分層
   單一專案的背景、規格與偏好，應與個人的長期工作偏好分開管理，避免混淆與污染。

5. 可演化優先於一次定完
   第一版不需要完整，但結構要能支援之後逐步擴充，不要一開始就做成笨重、難修改的大系統。

6. 核心知識共用，各工具只負責接入與使用
   ChatGPT、Gemini、VS Code、Antigravity 都只是工作入口。真正重要的知識應放在共用核心裡，各工具只需要用最少的方式接進來使用。

7. 沉澱要有門檻
   不是每段對話都值得進知識庫。只有重複出現、明顯重要、或確實能提升後續品質的內容，才應該被提取與固化。

8. 可驗證優先於感覺良好
   每個設計都應該能回答它是否真的降低 prompt 長度、提高首輪可用率、減少重複說明，或加快新專案啟動。

## 反原則

- 不要做成需要大量手動維護的知識庫
- 不要做成只適用單一工具的封閉系統
- 不要把所有對話都原封不動塞進記憶層
- 不要為了完整而犧牲使用頻率
- 不要讓 SKILL 數量膨脹到難以選擇

## 對話紀錄可提取的內容

- 常見任務類型
- 偏好規則
- 常修正 AI 的地方
- 喜歡的輸出格式
- 專案背景與術語
- 決策理由
- 可升級成 SKILL 的高頻模式

## 接下來要討論的內容

### 第一階段：原則

要先定義這套系統應該遵守的設計原則，例如：

- 低摩擦優先
- 可持續累積優先
- 跨工具重用優先
- 人工可覆核優先
- prompt 應越來越少，而不是越來越多

### 第二階段：策略

要決定採用哪種推進路線，例如：

- 先從單一工具開始，還是直接多工具整合
- 先做知識沉澱，還是先做執行助手
- 先做手動、半自動、或全自動流程

## 策略定版

### 1. 用跨工具思維設計，但 MVP 先從單一工具開始

- 架構上以跨 ChatGPT、Gemini、VS Code、Antigravity 共用核心知識為前提
- 實作上先從最容易取得資料、最容易驗證效果的工具開始
- 第一個資料來源優先選擇 VS Code / Copilot Chat

### 2. 第一優先做知識沉澱，不先做執行助手

- 先建立對話紀錄到結構化知識的沉澱流程
- 讓後續任何 agent 或 workflow 都有可依賴的記憶層
- 執行助手與自動化行為放在後續階段

### 3. 第一階段以專案為核心

- 先讓每個專案擁有自己的背景、規格、偏好、決策與常見模式
- 等多個專案累積後，再往上提煉成任務類型模板與通用 workflow

### 4. 第一階段採半自動模式

- 系統先提議哪些內容值得沉澱
- 經人工確認後再寫入知識庫或專案文件
- 先確保品質與可控性，再逐步提高自動化程度

### 5. 先追求可持續使用，再追求高智慧化

- 第一優先是讓系統真的進入日常工作
- 第二優先才是增加更多 agent、推理與自動判斷能力
- 任何設計都不應為了「看起來更聰明」而犧牲穩定性與使用頻率

### 第三階段：運作方式

要定義系統平常怎麼運作，例如：

- 何時擷取對話
- 何時萃取知識
- 何時更新文件
- 何時建立 SKILL
- 哪些變更需要人工確認

### 第四階段：MVP

只驗證最小閉環，不做大而全。可能的 MVP 方向：

- 先讀取 VS Code 對話紀錄
- 萃取偏好與 task patterns
- 更新專案記憶文件
- 在下一輪對話中驗證是否真的降低啟動摩擦

## 文件結構方向

第一階段採用：

- 每個專案一個資料夾
- 資料夾內按主要類型拆檔
- 保留一份入口總覽檔作為人與 AI 的共同入口

建議結構：

```text
project-name/
  README.md
  project-context.md
  preference-rules.md
  task-patterns.md
  output-patterns.md
  decision-log.md
  skill-candidates.md
```

各檔案角色：

- `README.md`: 專案入口與快速總覽
- `project-context.md`: 專案背景、限制、術語與核心脈絡
- `preference-rules.md`: 使用者偏好、禁忌、判斷標準
- `task-patterns.md`: 專案內高頻任務流程
- `output-patterns.md`: 偏好的交付格式與表達方式
- `decision-log.md`: 已定案事項與決策理由
- `skill-candidates.md`: 值得升級為 SKILL 或固定 workflow 的模式

採用這種結構的原因：

- 比單一大檔更容易長期維護
- 比過度細分更容易開始使用
- 容易支援半自動提取與寫入
- 有利於未來跨專案抽出共通 workflow 與 shared knowledge

後續整體結構可再往下延伸為：

```text
ai-system/
  personal/
  projects/
  shared/
```

但第一階段先以 `projects/` 內的專案記憶為核心，不先展開太多層次。

## 第一階段 MVP

第一階段 MVP 只驗證一條最小閉環：

`讀取 VS Code 對話紀錄 -> 提取可沉澱內容 -> 寫入專案記憶檔 -> 下一次對話前先讀取 -> 觀察是否更貼近需求`

### MVP 範圍

第一階段只做以下事項：

1. 只處理 `VS Code / Copilot Chat` 的對話紀錄
2. 優先提取以下 4 類內容：
   - `project-context`
   - `preference-rules`
   - `task-patterns`
   - `decision-log`
3. 提取結果寫入專案資料夾中的 markdown 檔案
4. 新一輪對話前，先讀取專案入口檔與相關記憶檔
5. 在工作後提出可沉澱項目，由人工確認後寫入

### 第一階段先不做

- 多工具同步
- 自動寫入所有對話
- 自動建立 SKILL
- 個人層知識的全自動更新
- 複雜搜尋介面
- 向量資料庫或 embedding
- 過度細緻的 metadata 系統

### MVP 成功標準

- 同一專案再次啟動時，需要補充的背景資訊明顯下降
- AI 首輪回覆更接近既有偏好與專案脈絡
- 已定案的重要決策不需要重複說明
- 使用者願意持續使用這套流程，而不是回到原本零散的提示詞方式

## MVP 角色分工

第一階段先只定義 3 個角色：

### 1. 使用者

負責：

- 提供任務與對話內容
- 確認哪些內容值得沉澱
- 核准正式寫入專案記憶

### 2. 提取助手

負責：

- 讀取對話紀錄或指定材料
- 找出可沉澱的內容
- 判斷應歸入哪一類記憶
- 產出待確認的提取結果

### 3. 專案記憶維護者

負責：

- 將已確認內容寫入對應 markdown 檔
- 維持檔案結構與內容一致性
- 避免重複、衝突或錯誤覆寫

第一階段中，「提取助手」與「專案記憶維護者」可以由同一個 AI agent 扮演，不必強制拆成兩個實體。

## MVP 流程

### 流程 1：工作前

1. 指定專案
2. 讀取 `README.md` 與相關專案記憶檔
3. 用已知專案脈絡開始新一輪任務

### 流程 2：工作中

1. 使用者與 AI 在專案脈絡下合作
2. AI 優先依照既有記憶與規則回應
3. 若出現新的背景、偏好、流程或決策，先保留在對話中，不急著即時寫入

### 流程 3：工作後

1. AI 提出本輪值得沉澱的內容
2. 使用者確認、刪除或修改提取結果
3. 將確認後的內容寫入對應檔案
4. 若某模式已反覆出現，可標記為 `skill-candidate`

## MVP 觸發時機

第一階段先採明確觸發，不做背景自動常駐：

- 開始專案任務前，主動讀取專案記憶
- 某次對話結束後，主動進行沉澱整理
- 當重複模式明顯出現時，再手動決定是否升級為 SKILL 候選

## 第一版沉澱規則

第一階段的重點不是「盡量多存」，而是「只存真正有重用價值的內容」。

### 一、值得沉澱的內容

符合以下任一條件，就應列為沉澱候選：

- 對同一專案之後的工作有持續影響
- 已明確形成決策，之後不希望重複討論
- 反覆出現兩次以上，具有穩定模式
- 能明顯降低下一次對話的說明成本
- 能明顯提升下一次產出的品質與貼合度

### 二、不值得沉澱的內容

符合以下情況，原則上不寫入正式記憶：

- 只在當次情境成立的一次性指令
- 沒有後續重用價值的臨時想法
- 只是聊天過程中的措辭變化
- 尚未確認、仍在探索中的零散假設
- 純粹工具操作細節，但不具長期參考價值

### 三、分類判斷規則

#### 寫入 `project-context` 的條件

當內容屬於以下類型時，應歸入 `project-context`：

- 專案目標與範圍
- 背景脈絡與已知事實
- 重要術語與定義
- 專案限制、限制來源與固定前提
- 利害關係人、對象、情境描述

#### 寫入 `preference-rules` 的條件

當內容屬於以下類型時，應歸入 `preference-rules`：

- 使用者明確表示喜歡或不喜歡某種做法
- 對語氣、格式、深度、結構有穩定偏好
- 對工作方式有固定習慣或禁忌
- 對「什麼算好結果」有明確判準

補充門檻：

- 只出現一次的偏好，先列為候選
- 出現兩次以上，或被明確確認後，再正式寫入

#### 寫入 `task-patterns` 的條件

當內容屬於以下類型時，應歸入 `task-patterns`：

- 某類任務有固定步驟
- 某類任務反覆需要相似的輸入
- 某類任務反覆產出相似結果
- 已能辨識出可重複的流程骨架

補充門檻：

- 單次成功做法先保留觀察
- 至少重複兩次以上，再視為穩定模式

#### 寫入 `output-patterns` 的條件

當內容屬於以下類型時，應歸入 `output-patterns`：

- 使用者反覆要求特定交付結構
- 某類輸出有穩定章節順序
- 使用者對篇幅、語氣、細節層次有固定要求

補充門檻：

- 必須具備跨多次任務的重用價值

#### 寫入 `decision-log` 的條件

當內容屬於以下類型時，應歸入 `decision-log`：

- 已明確定案
- 對後續做法有影響
- 有必要保留決策理由，避免日後回頭重議

補充門檻：

- 沒定案的不記
- 小型、無後續影響的微調不記

### 四、升級為 `skill-candidate` 的門檻

只有同時符合多項條件，才列為 `skill-candidate`：

- 已重複出現至少三次
- 有穩定的輸入與輸出形式
- 中間步驟具備可描述性
- 判斷標準相對清楚
- 明顯值得標準化，能降低後續摩擦

### 五、人工確認規則

第一階段以下內容都需要人工確認後才能正式寫入：

- 新的 `preference-rules`
- 新的 `decision-log`
- 所有 `skill-candidate`
- 任何可能影響後續大量任務的規則變更

### 六、簡化判斷原則

若提取時無法快速判斷，優先採以下原則：

- 不確定是否值得沉澱時，先不寫正式記憶
- 不確定分類時，先放候選區而不是硬分類
- 不確定是否為長期偏好時，先觀察下一次是否重複出現

## 第一版提取分類

第一階段先只提取 6 類內容，避免分類過細而失去可用性。

### 1. project-context

用途：

- 記錄專案背景、目標、限制、術語、關係人與既有脈絡

應包含：

- 專案目的
- 重要術語
- 關鍵限制
- 已知背景事實
- 需要持續參照的上下文

不應包含：

- 臨時的單次指令
- 純粹一次性的聊天噪音

### 2. preference-rules

用途：

- 記錄使用者穩定偏好、寫作標準、格式偏好、工作習慣與禁忌

應包含：

- 偏好的語氣與表達方式
- 不喜歡的說法
- 喜歡的輸出結構
- 決策時的判準
- 固定的工作習慣

不應包含：

- 只在單次情境成立的要求
- 尚未穩定重複出現的個人偏好猜測

### 3. task-patterns

用途：

- 記錄某類任務通常怎麼展開、有哪些步驟、常見輸入與輸出

應包含：

- 任務名稱
- 常見前置資訊
- 執行步驟
- 產出形式
- 檢查點

不應包含：

- 太細碎、只適用一次的動作
- 純粹工具操作細節但沒有重用價值的內容

### 4. output-patterns

用途：

- 記錄使用者偏好的交付形式與可重用的結果樣式

應包含：

- 結構模板
- 常見段落順序
- 語氣風格
- 詳略程度
- 呈現偏好

不應包含：

- 單篇內容本身
- 沒有重用價值的臨時格式

### 5. decision-log

用途：

- 記錄重要決策與決策理由，避免日後重複討論或失去脈絡

應包含：

- 決策內容
- 決策日期
- 決策理由
- 影響範圍
- 若有，列出放棄的替代方案

不應包含：

- 還未真正定案的零散想法
- 沒有後續影響的小型聊天判斷

### 6. skill-candidates

用途：

- 記錄值得升級成 SKILL、固定 workflow 或 agent 規則的高頻模式

應包含：

- 重複出現的任務模式
- 穩定的輸入輸出結構
- 清楚的判斷規則
- 可以被標準化的流程

不應包含：

- 還在摸索中的做法
- 沒有穩定重複證據的臨時靈感

## 文件維護方式

這份文件作為目前的總規劃文件。後續每次定案時：

- 將已決定的內容補進對應章節
- 保留尚未定案的選項與問題
- 視需要再拆出更細的子文件
