# 2026-03-28 roadmap 單一真源與全貌化建議

## 目的

這份文件用來回答三個問題：

- `docs/roadmap.md` 應該扮演什麼角色
- V2 做完後，這個專案是否就算完成
- roadmap 應該如何同時呈現「當前進度」與「整個專案全貌」

## 結論摘要

### 結論 1：`docs/roadmap.md` 應該成為專案進度的單一真源

接下來專案的進度判讀，應優先以 `docs/roadmap.md` 為準。

它應該回答：

- 這個產品的長期目標是什麼
- 現在位於哪一個版本階段
- 當前版本做到哪裡
- 下一個版本是什麼
- 還有哪些更長期的版本規劃

這代表：

- `docs/handoff/current-task.md` 應只保留短期交接與下一步
- `docs/product/user-guide-current.md` 應只描述現在可用能力
- 版本完成度、Phase 狀態、下一階段方向，都應先以 `docs/roadmap.md` 更新

### 結論 2：V2 做完後，不代表整個專案完成

如果回到這個專案最初的目標，V2 做完後仍不算真正完成。

原始目標包含：

- 核心知識庫
- 專案記憶持續累積
- 低摩擦 handoff
- 對話提取與知識沉澱
- 跨工具使用
- 長期形成個人工作系統

依這個範圍來看：

- `V1` 應該是單專案可用的知識閉環工作台
- `V2` 應該是更可靠、更穩定、可管理多專案的工作台
- `V3` 應該是跨工具整合與標準化輸入
- `V4` 應該是治理、自動化與更完整的個人 AI 工作作業系統

所以答案是：

**V2 不是終點。至少還應該有 V3、V4。**

### 結論 3：roadmap 現在最大問題不是「有沒有內容」，而是「層次混在一起」

目前 `docs/roadmap.md` 已經有：

- 產品願景
- 實作 Phase
- 規格層 archive
- 異動記錄

但它還缺一個更清楚的分層：

1. 長期產品全貌
2. 當前版本計畫
3. 當前版本進度
4. 歷史 archive

如果這四層沒分開，讀 roadmap 時很容易出現兩個問題：

- 只看得到現在的 Phase，看不到之後的 V3、V4
- 只看得到遠景，不知道現在到底做到哪

## 建議的 roadmap 角色定義

我建議 `docs/roadmap.md` 以後固定承擔以下責任：

### 1. 專案全貌入口

讀者打開 roadmap，就能知道這個產品不是只有目前這一版，而是有完整版本路線。

### 2. 目前版本進度真源

所有「現在做到哪裡」的判斷，都先以 roadmap 為準。

### 3. 版本之間的橋接文件

roadmap 應同時說明：

- 現在版本在做什麼
- 下一個版本要解什麼問題
- 這些版本如何一路接回最初目標

### 4. 文件治理基準

其他文件若要描述進度，應引用 roadmap，而不是自行維護另一套版本狀態。

## 建議的 roadmap 結構

我建議把 `docs/roadmap.md` 改成下列結構。

### A. 願景與北極星

一句話回答這個產品最後想變成什麼。

例如：

> 從 markdown 驅動的單專案工作流，成長為跨工具、可持續累積知識、可協助交接與沉澱的個人 AI 工作台。

### B. 版本全貌地圖

這一段專門回答：V1、V2、V3、V4 各自是什麼。

建議用表格呈現：

| 版本 | 版本定位 | 做完後你可以… | 核心能力 |
|------|----------|----------------|----------|
| V1 | 單專案知識閉環工作台 | 在單一專案中完成 handoff、提取、審核、寫回、檢視 | 基礎閉環 |
| V2 | 穩定化與多專案工作台 | 更安全地寫回、多專案切換、工作台更可信任 | 穩定化與擴展 |
| V3 | 跨工具整合層 | 接入 ChatGPT、Gemini、VS Code、Antigravity 等多種來源 | 多工具標準化 |
| V4 | 個人 AI 工作作業系統 | 有治理、自動化、共享規則與更完整的工作習慣沉澱 | 自動化與治理 |

### C. 當前版本執行計畫

這一段只談「現在正在做的版本」。

例如如果現在仍在收斂 V1，這一段就只列：

- V1 的目標
- V1 的 Phase
- V1 目前做到哪裡
- V1 下一步

### D. 當前狀態摘要

這一段應用非常白話的方式說清楚：

- Current release：現在是哪個版本
- Current status：現在做到哪個里程碑
- Next milestone：下一個里程碑是什麼
- Known gaps：目前還差什麼

### E. 驗收與升級條件

這一段回答「什麼情況下 V1 可以升成 V2、V2 可以升成 V3」。

例如：

- V1 不只要頁面存在，還要知識閉環可用
- V2 不只要有 projects 頁，還要是真正的多專案資料切換
- V3 不只要匯入檔案，還要有統一的 conversation schema 或 adapter

### F. 異動記錄

保留現在的異動記錄區塊，但只記重要路線調整與版本狀態變更。

## 建議的版本定義

以下是我建議放進 roadmap 的版本地圖。

## V1：單專案知識閉環工作台

### 版本目標

把這個專案做成一個「單專案可用」的工作台，能讓使用者：

- 讀取專案現況
- 產生 handoff
- 從對話提取候選
- 審核與寫回記憶
- 檢視決策與規則

### 核心價值

- 降低新對話摩擦
- 把一次性對話變成可重用資產

### 完成標準

- handoff 可真正使用
- extraction + review + writeback 可真正跑通
- Memory / Decisions / Rules 能反映寫回結果
- roadmap、handoff、current guide 對狀態描述一致

## V2：穩定化與多專案工作台

### 版本目標

讓這個產品從「可用雛形」變成「可信任的工作台」。

### 核心能力

- writeback 更安全
- 真正多專案切換
- 更完整但仍輕量的 UI
- 更好的 review 與搜尋

### 建議納入範圍

- backup / draft / merge 策略
- project data source 真切換
- Projects Hub 反映不同專案
- 更清楚的 loading / empty / error 狀態

## V3：跨工具整合層

### 版本目標

讓這套工作台不只服務單一入口，而能接住多個 AI 工具產生的對話與知識。

### 核心能力

- ChatGPT / Gemini / VS Code / Antigravity 對話來源整合
- conversation schema 標準化
- importer / adapter 機制
- shared / personal / project 層級的知識分流

## V4：治理、自動化、個人作業系統

### 版本目標

讓這個產品從「工作台」再往上走，變成真正持續幫助使用者工作的個人 AI 作業系統。

### 核心能力

- 自動或半自動治理流程
- 例行工作自動化
- 記憶品質檢查
- 規則衝突偵測
- 跨專案共享模式與技能沉澱

## 對 `docs/roadmap.md` 的具體修改建議

### 建議 1：刪除或改寫「這就是全部。沒有隱藏的 Phase。」

如果 roadmap 要承載整個專案全貌，這句話不適合保留。

應改成：

- V1 有明確邊界
- 但產品整體仍有 V2、V3、V4 路線

### 建議 2：把「版本地圖」和「當前執行 Phase」拆開

不要讓讀者把：

- `V1 的 Phase 1–5`

和

- `整個產品只有 Phase 1–5`

混為一談。

### 建議 3：明確標出「Current Release」

例如：

- Current release: V1
- Release status: 核心閉環已具雛形，進入校準 / 穩定化

這比單純寫「全部完成」更準確，也更不容易誤導後續 agent。

### 建議 4：讓 roadmap 成為唯一進度判讀入口

建議新增一段治理規則，明確寫入：

- 版本完成度先更新 `docs/roadmap.md`
- `docs/handoff/current-task.md` 只記錄短期交接，不再單獨宣告整體版本完成
- `docs/product/user-guide-current.md` 只描述已交付能力，並以 roadmap 當前版本狀態為準

### 建議 5：用「版本升級條件」取代過度樂觀的完成宣告

比起直接寫某版本完成，更穩的做法是明確列出：

- 還缺哪些條件才算真的完成
- 哪些功能目前只是 beta / partial

## 我對目前專案狀態的建議表述

如果只用一句話描述現在的專案狀態，我建議：

> 專案已做出 V1 核心工作台骨架與知識閉環雛形，但仍需要以 roadmap 為單一真源重新校準版本狀態，並補出 V2–V4 的產品全貌。

## 最後結論

這個專案的 roadmap 不應只回答「今天做到哪」，
也必須回答「這個產品最後會走到哪」。

所以：

- `roadmap.md` 要變成單一真源
- `roadmap.md` 要看得到 V1、V2、V3、V4
- `roadmap.md` 要同時呈現長期全貌與當前進度

這樣後續不論是你自己、coding agent，還是其他 AI 工具接手，都能從同一份文件看懂專案全貌與現在位置。
