# Update Workflow

## 目的

定義本專案在日常使用時，專案記憶應如何被讀取、提取、確認與更新。

目標不是增加流程負擔，而是用最小成本讓對話成果逐步沉澱成可重用知識。

## 基本原則

- 工作前先讀，工作後再沉澱
- 不在工作中途頻繁改檔
- 不確定的內容先列為候選，不直接寫成正式規則
- 只有會影響之後工作的內容才進正式記憶

## 工作前流程

### 必讀檔案

開始專案任務前，至少先讀：

- `README.md`

依任務需要補讀：

- `project-context.md`
- `preference-rules.md`
- `task-patterns.md`
- `output-patterns.md`
- `decision-log.md`

### 讀取規則

- 如果是新任務或方向型討論，優先補讀 `project-context.md` 與 `decision-log.md`
- 如果是要產出內容，優先補讀 `preference-rules.md` 與 `output-patterns.md`
- 如果是重複性工作，優先補讀 `task-patterns.md`

## 工作中流程

- 優先依照既有專案記憶合作
- 若出現新背景、新偏好、新流程、新決策，先留在對話中，不急著即時寫檔
- 若工作中發現既有規則有衝突，先標記問題，工作後再整理

## 工作後流程

### 第一步：提出沉澱候選

工作告一段落後，先整理本輪值得沉澱的內容：

- 新的專案背景
- 新的偏好規則
- 新的任務模式
- 新的輸出模式
- 新的決策
- 新的 skill 候選

### 第二步：人工確認

逐項確認：

- 是否真的有重用價值
- 是否會影響後續任務
- 是否屬於長期內容而非一次性內容
- 是否分類正確

### 第三步：寫入對應檔案

確認後依內容類型更新：

- 專案背景 -> `project-context.md`
- 偏好規則 -> `preference-rules.md`
- 任務模式 -> `task-patterns.md`
- 輸出模式 -> `output-patterns.md`
- 已定案事項 -> `decision-log.md`
- 可標準化候選 -> `skill-candidates.md`

必要時同步更新：

- `README.md`
- `ai-workflow-planning.md`

## 更新判斷速查

### 應更新 `README.md` 的情況

- 專案狀態有明顯變化
- 重要檔案新增或角色改變
- 工作方式需要新的入口說明

### 應更新 `project-context.md` 的情況

- 專案目標、背景、限制、術語有新增或修正
- 有新的關鍵脈絡會影響後續理解

### 應更新 `preference-rules.md` 的情況

- 使用者明確確認某項偏好或禁忌
- 某偏好已重複出現並被驗證

### 應更新 `task-patterns.md` 的情況

- 某類工作已出現穩定步驟
- 某流程已重複出現至少兩次

### 應更新 `output-patterns.md` 的情況

- 某種交付結構或語氣偏好已穩定出現

### 應更新 `decision-log.md` 的情況

- 有明確定案
- 該決策會影響之後的做法

### 應更新 `skill-candidates.md` 的情況

- 某模式已出現標準化價值
- 該模式具備穩定輸入、輸出與步驟

## 建議節奏

- 每次重要對話後做一次簡短沉澱
- 每完成一個階段性成果後做一次整理
- 不需要每一輪短對話都更新檔案

## 第一階段建議做法

第一階段先用最簡單方式執行：

1. 開始任務前讀取相關檔案
2. 完成任務後整理沉澱候選
3. 經人工確認後更新對應 markdown
4. 若有新的工作習慣或流程，再回頭修正這份文件
