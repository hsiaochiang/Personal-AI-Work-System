# Conversation And Branch Strategy

## 目的

定義本專案在規劃、實作、分支管理與對話切分上的基本策略，降低長對話帶來的漂移與幻覺，同時避免開出過多難以管理的分支。

## 核心原則

- 用文件承接長期記憶
- 用短對話處理單次任務
- 用 git 分支隔離程式變更
- 不用 git 分支來解決對話過長問題

## 一、對話串角色分工

### 1. 規劃串

用途：

- 定義目標
- 定義原則
- 定義策略
- 定義運作方式
- 做重要決策

特徵：

- 可以較長
- 以方向與判斷為主
- 不適合承載大量具體改檔工作

### 2. 實作串

用途：

- 完成單一明確變更
- 查 codebase
- 改檔
- 驗證與修正

特徵：

- 一串一個明確目標
- 生命週期短
- 驗收標準明確

### 3. 整合串

用途：

- 回收多個實作結果
- 檢查是否與既有決策衝突
- 更新專案記憶
- 準備合併

特徵：

- 不做大量新實作
- 以整合、確認與回寫為主

## 二、何時應開新對話

符合以下條件之一，就應優先考慮開新對話：

- 本次任務目標已和原對話不同
- 需要讀取的上下文已明顯切換
- 原對話已累積過多歷史，開始影響精準度
- 本次工作可以用短 handoff 說清楚

不需要開新對話的情況：

- 仍在同一目標下做小幅續作
- 只是補一個小修正或小測試
- 沒有明顯上下文切換

## 三、何時應開新分支

符合以下條件之一，才建議開新分支：

- 變更內容獨立
- 涉及的檔案群相對獨立
- 可以單獨驗證
- 短期內不需要與其他變更交錯進行

不建議只因為以下原因開分支：

- 對話變太長
- 想讓 AI 重新整理上下文
- 同一功能仍在連續迭代

## 四、推薦結構

本專案採用 Hub-and-Spoke 結構：

- 一個主規劃串作為 hub
- 多個短生命週期實作串作為 spoke
- 規劃與決策回寫到專案文件
- 實作串只讀取必要文件與 handoff，不背整段歷史

## 五、每個實作串的最小 handoff

每個新實作串開始前，應先準備最小 handoff，至少包含：

- 這次要做什麼
- 這次不要做什麼
- 已定案原則
- 相關檔案
- 驗收標準
- 前一串做到哪裡

## 六、工作流建議

### 開始前

1. 確認這次是規劃、實作還是整合
2. 若是實作，先確認是否真的需要新分支
3. 準備最小 handoff
4. 讀取必要的專案記憶檔

### 進行中

1. 對話只聚焦在單一目標
2. 若範圍開始膨脹，先停下來收斂
3. 若發現新決策，先記錄，完成後再回寫

### 結束後

1. 整理本次做了什麼
2. 回寫到 `decision-log.md`、`task-patterns.md` 或其他對應檔案
3. 若形成穩定模式，再加入 `skill-candidates.md`

## 七、實務判斷句

若遇到是否要切分的猶豫，可用以下判斷：

- 如果只是上下文太長，但工作目標沒變：換對話，不一定換分支
- 如果目標變了，但程式碼邊界沒變：通常換對話，不一定換分支
- 如果變更邊界獨立且可單獨驗證：可考慮開新分支
- 如果只是想讓 AI 不要漂：先準備 handoff，而不是先開分支

## 八、目前採用的結論

- 保留規劃串與實作串分離的做法
- 實作時優先以「一串一目標」運作
- 分支以程式變更邊界為準，不以對話長度為準
- 每次實作完成後，應回寫專案記憶，避免知識留在單一對話內
