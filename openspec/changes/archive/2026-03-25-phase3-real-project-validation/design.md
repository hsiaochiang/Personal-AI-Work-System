# Design: phase3-real-project-validation

## Context

S2 已完成規格化的半自動提取流程，但成功條件主要集中在流程可執行與文件可驗證。
S3 需要回答的是：在真實專案工作負載下，候選是否可用、去重是否可靠、治理是否可持續。
因此本設計採驗證導向而非功能擴張策略，避免在證據不足時提前投入 UI 或架構演進。

## Goals / Non-Goals

Goals:
- 在真實專案情境完成至少 2 次端到端驗證 run。
- 對候選可用性與 dedupe 結果建立可量測標準。
- 確保治理證據可追溯、可交接、可重放。
- 明確鎖定 S2 non-regression，不回頭擴改 S2 scope。

Non-Goals:
- 不做 UI 設計或互動流程開發。
- 不改動 S2 核心契約與欄位定義。
- 不整合外部工具來源或新增資料平台。
- 不追求全自動化決策與回寫。

## Decisions

1. 驗證樣本採真實任務來源，不接受純合成資料作為唯一驗證依據。
- 每次 run 必須引用真實專案工作上下文（任務討論、變更規劃或實作交接內容）。
- 若含合成補充資料，必須明確標註且不得取代真實樣本。

2. 採雙 run 最小驗證門檻。
- 至少 2 次獨立 run，避免單次偶然性。
- 每次 run 均需完整記錄輸入、候選、審核、回寫或拒絕、時間與操作者。

3. 候選品質以人工可採納率為主指標。
- approved 候選中可直接採納率需達 70% 以上。
- 低於門檻時需記錄原因分類（抽象過度、證據不足、重複、非行動化）。

4. dedupe 衝突採顯式處理。
- 所有 dedupe_key 衝突必須有處理結果（合併、保留、拒絕）。
- 未處理衝突數不得大於 0。

5. 治理同步列為 release gate。
- strict validate PASS 與治理文件同步視為同級 gate。
- 任一 gate 未通過不得宣稱 S3 完成。

## Risks / Trade-offs

- 風險：真實樣本數不足，指標代表性偏弱。
  緩解：至少 2 次 run，且鼓勵跨不同任務情境取樣。

- 風險：驗證紀錄成本提高，執行負擔上升。
  緩解：採最小必要欄位模板，避免過度文書化。

- 風險：S3 驗證過程被拉向新功能開發。
  緩解：tasks 設 non-scope gate，超出項目只記錄為後續候選，不在本 change 實作。

- 風險：CLI 指令演進造成執行混淆。
  緩解：明確保留主指令與備援指令，並在 runlog 記錄實際採用命令。

## Migration Plan

- 無資料庫 migration。
- 僅新增或更新 OpenSpec 與治理文件，採 docs-first 方式推進。
- 若 S3 驗證結果不達門檻，僅輸出修正建議，不直接擴 scope 實作。

## Open Questions

- S3 驗證完成後，S4 的收斂定版是否需新增固定品質儀表欄位。
- 候選可採納率 70% 門檻是否需依任務類型做分層。
